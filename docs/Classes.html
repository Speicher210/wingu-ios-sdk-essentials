There will never be a need to use more than one instance of BeaconScanner to work with wingu beacons.   • Beacon

    Representation of CoreLocation class CLBeacon with extra arguments from wingu portal.
    This class combines fetched arguments from wingu portal such as Deck, Pack, Content and match it with physical characteristic of the beacon. All parameters from CLBeacon will be matched with the same naming to this class.

    Note: This is not extend or override any CLBeacon parameters. Any changes in object of this class will not take any effect in physical beacon!

    This class is marked as final, this shouldn’t need subclassed. If there is a need to change minor/major/uuid of physical beacon this should be done with beacons manufacturer dedicated application or SDK.

    See more

  • Channel

    Base representation of some wingu trigger configured in portal. All detailed triggers like Geofence or Beacon will inherit from this class.
    Detailed information about triggers can be found in class specific for every type. This class contain parameters related to wingu portal or time of discovery.

    Note: All delegates method will always return Channel or collection of Channel even if you monitor only for one type.

    However object of this class without subclass cannot be used for anything inside wingu SDK. Initialization should be avoided and all method will map to one of the supported subclass.

    Pack collection for multilanguage is not supported with current version of the SDK. It will be returned always with end-user device language whenever your configured content on wingu portal supports it.

    See more
